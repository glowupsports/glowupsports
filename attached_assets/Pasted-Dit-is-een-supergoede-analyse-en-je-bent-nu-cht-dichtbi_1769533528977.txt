Dit is een supergoede analyse — en je bent nu écht dichtbij “bulletproof”. Wat je mist is niet nóg een flow, maar één harde contractlaag die ALLE 4 attendance-paths verplicht dezelfde credit-logica laat volgen + een paar database-guards zodat je nooit dubbel kan afschrijven.

Hier is hoe ik het zou “harden” zodat het nooit meer stuk gaat, ook niet als auto-attendance en coach attendance tegelijk draaien.

1) Maak één single entrypoint: ensureCreditProcessed(sessionPlayerId)

Nu roepen je 4 flows allerlei varianten aan (deductTypedCreditsForSession, consumeSingleCreditForSession, complete, auto, etc.). Dat is waar chaos ontstaat.

✅ Nieuwe regel:
ELKE plek die attendance verandert naar present/late moet maar één functie aanroepen.

Contract

ensureCreditProcessed(sessionPlayerId) doet:

lock / transactiestart

read session_players row FOR UPDATE

Als attendanceStatus niet in present|late → return (no-op)

Als creditDeductedAt != null OR creditTransactionId != null → return (idempotent)

probeer credit te consumeren (package → debit)

als geen package → maak debt (debt tx) en laat creditDeductedAt NULL

commit

Belangrijk: je idempotency guard moet niet alleen op creditDeductedAt vertrouwen. Gebruik óók creditTransactionId én een unieke constraint (zie punt 3).

2) Fix je “reason” chaos: normalizeer naar één canonical reason

Je hebt nu session_booking, session_join, etc. Dat is prima voor historie, maar slecht voor filters.

✅ Nieuwe regel voor nieuwe transacties:

bij echte consumption: reason = "session_consumed"

bij debt: reason = "session_debt"

bij settlement: reason = "session_settlement"

Legacy reasons laat je bestaan, maar filters in code moeten voortaan werken op:

amount < 0 AND packageId IS NOT NULL = consumption

amount < 0 AND packageId IS NULL AND reason in ("session_debt", legacy_debt...) = debt

3) DB-level “no double debit” guard (dit is je echte bulletproof)

Je guards in code helpen, maar concurrentie (auto job + coach) kan alsnog dubbel gaan.

Je hebt 2 opties (ik raad A aan):

A) Voeg sessionPlayerId column toe aan credit_transactions

Dan kun je hard afdwingen:

UNIQUE(sessionPlayerId, reason) waar reason in consumption types

Als je geen partial unique index wil doen:

maak eventKey string veld (bv "consume:<sessionPlayerId>")

UNIQUE(eventKey)

B) Geen schema change? Dan unique op (playerId, sessionId, creditType, amount)

UNIQUE(playerId, sessionId, creditType, amount)
Maar dit is iets minder netjes.

Dit ene ding voorkomt 99% van “WTF waarom -2?!”

4) Attendance flows: wat ze mogen doen (en wat niet)
Coach mark attendance

Mag alleen:

attendanceStatus setten

lateMinutes setten

daarna: ensureCreditProcessed() per affected session_player

Auto-attendance job

Mag alleen:

ontbrekende session_players aanmaken (als nodig)

status op present zetten

daarna ensureCreditProcessed()

Complete session endpoint

Mag alleen:

missing attendance invullen (present)

daarna ensureCreditProcessed()

Join session endpoint

Mag alleen:

session_players aanmaken met pending

NOOIT credits verwerken

➡️ Als je dit afdwingt, maakt het niet uit hoeveel routes je hebt.

5) Debt & settlement: maak het “non double” en simpel

Je hebt al een beter nieuw ding: settleUnpaidSessions() (sessies met creditDeductedAt = null).

Ik zou het zo doen:

5.1 Deprecate settlePlayerDebts() of laat hem intern settleUnpaidSessions() callen

Want jouw “debt-transactions zoeken” is kwetsbaarder dan “sessies zonder creditDeductedAt”.

5.2 Settlement invariant

Als settlement gebeurt:

je maakt exact 1 consumption transactie met packageId

je zet session_players.creditDeductedAt + creditTransactionId

als er een debt transactie bestaat:

markeer hem settled via metadata (niet meetellen)

6) Wat je nu meteen kunt checken waarom je -9 ziet i.p.v. -15

Dit is de snelste debug die bijna altijd de oorzaak onthult:

Vraag 1: hoeveel session_players hebben present|late maar creditDeductedAt IS NULL?

Als dat 6 zijn → daarom zie je -9.

Vraag 2: zijn er consumptions die bestaan maar niet gelinkt zijn aan session_players?

(transactie met sessionId maar session_players.creditTransactionId null)
Dan heeft je UI/summary mismatch.

Vraag 3: heb je packages die “active” filteren en daardoor credits missen?

Bijv. status verkeerd (depleted/expired) waardoor deduct niet gebeurt.

7) De “repair” strategy die bij jouw model past

Omdat jij stored remainingCredits wilt vertrouwen (nu), moet je 2 dingen repareren:

Re-link session_players naar credit_transactions

voor elke session_player waar creditDeductedAt IS NOT NULL maar creditTransactionId IS NULL

zoek tx op sessionId+playerId+amount=-1+packageId not null → link hem

Re-process missing deductions

voor elke session_player met attendance present/late en creditDeductedAt IS NULL

run ensureCreditProcessed(sessionPlayerId) (idempotent)
