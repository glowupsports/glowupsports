Ja — dit kán absoluut de oorzaak zijn, maar met één belangrijke nuance:

Wanneer is dit écht het probleem?

Dit gaat over je DATABASE_URL (Postgres connectie). Dus dit is alleen de boosdoener als:

✅ jouw mobile app (Play Store build) praat met jouw eigen backend (Replit/Cloud Run)
➡️ en die backend praat met Supabase Postgres via SUPABASE_DATABASE_URL
➡️ en die backend crasht / kan niet connecten door die pooler/poort

Dan lijkt het in de app alsof “Supabase niet connect”, maar in werkelijkheid is je backend dood of time-out.

❌ Maar: als je app direct Supabase gebruikt via SUPABASE_URL + ANON_KEY (supabase-js in React Native), dan heeft poort 5432/6543 niets te maken met je probleem. Dan kijk je eerder naar env vars, keys, RLS, of auth redirect config.

Waarom 5432 vs 6543 bij serverless wél vaak breekt

Session pooler (5432) = bedoeld voor lange sessies / stateful verbindingen. In serverless kan dat instabiel zijn (veel cold starts, korte verbindingen, scaling).

Transaction pooler (6543) = bedoeld voor serverless / korte bursts → meestal de juiste keuze.

Die waarschuwing “Only use on an IPv4 network” is precies zo’n red flag: sommige runtimes of routes doen (deels) IPv6 of hebben NAT/egress gedoe → dan faalt je connectie random of altijd.

Dus als jouw backend serverless draait: switch naar 6543 is vaak meteen fix.

Concrete fix (doe dit)

Update je secret SUPABASE_DATABASE_URL naar Transaction pooler :6543

Redeploy je backend (Replit deploy/Cloud Run)

Test backend health (heel belangrijk):

open je backend URL in browser: /health of een simpele endpoint

of check logs: “connected to database” / “pool created”

Test daarna je app opnieuw.

Let op: je URL die je stuurde heeft waarschijnlijk een kleine typo (die bracket). Het moet zo zijn (voorbeeld):

postgresql://postgres.<projectref>:<PASSWORD>@aws-1-<region>.pooler.supabase.com:6543/postgres

(Exacte host/region neem je uit Supabase dashboard, niet gokken.)

Supersnelle check: is het client-side of backend-side?

Zeg even heel kort welke van de twee jij doet:

A) App → Supabase direct (supabase-js)

Dan is dit niet de oorzaak. Dan moeten we checken:

EXPO_PUBLIC_SUPABASE_URL / EXPO_PUBLIC_SUPABASE_ANON_KEY in EAS production

RLS policies

Auth redirect / deep link config

B) App → jouw backend → database

Dan is dit heel waarschijnlijk wél de oorzaak, en 6543 is de juiste move.
