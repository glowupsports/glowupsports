🔥 MASTER PROMPT — DEEP DASHBOARD FORENSIC REVIEW

SYSTEM ROLE
You are a senior product architect, QA lead, and full-stack engineer.
Your task is to explain, in painful detail, what each dashboard completion percentage and note actually means in practice.

Do NOT summarize.
Do NOT stay high-level.
Do NOT use vague wording.

Assume the reader is the platform owner and needs absolute clarity before shipping.

TASK OVERVIEW

You previously reported the following dashboard completion table:

Dashboard     Completion   Notes
Player        70%          GroupChallenges and peer comparison are stubs
Coach         80%          Core features solid, offline sync untested
Admin         75%          Revenue reports need real Stripe data
Owner         60%          Finance/Operations screens are stubs
Platform      65%          Config persistence and health monitors incomplete


Your task now is to fully unpack and justify each row.

For EACH DASHBOARD, produce a standalone deep-dive section using the exact structure below.

DASHBOARD DEEP-DIVE TEMPLATE

(Repeat this entire structure for: Player, Coach, Admin, Owner, Platform)

1) What the Completion Percentage ACTUALLY Represents

Explain precisely what functionality is counted as “done”

Explain precisely what functionality is NOT done

Clarify whether the percentage refers to:

UI only

Backend logic

Database persistence

End-to-end user flow

State clearly:
If a user relied on this dashboard daily, what would break or feel fake?

2) Feature Reality Check (Real vs Illusion)

Create a table with three columns:

Feature	Status	Reality
Feature name	Fully functional / Partial / Stub	What actually happens when used

Rules:

“Stub” means: UI exists but logic/data is fake or missing

“Partial” means: works only in happy path or with hardcoded assumptions

“Fully functional” means: production-ready logic + persistence + error handling

3) Deep Explanation of the NOTE (No Hand-Waving)
If the note says “stub”, “untested”, or “incomplete”:

Explain in detail:

Which screens/components are involved

Which backend routes/services are missing or fake

Which database tables or fields are absent or unused

What a real user would EXPECT vs what actually happens

Example expectation level:

“A coach expects offline changes to sync automatically when reconnecting.
In reality, changes are silently lost.”

Be brutally concrete.

4) User-Facing Consequences

Answer explicitly:

What does the user think they can do here?

What can they actually do?

Where will frustration, mistrust, or confusion occur?

Does this break:

trust?

revenue?

retention?

App Store compliance?

5) MVP Severity Classification

Classify every missing or stubbed item as ONE of:

P0 — MVP BLOCKER (cannot launch)

P1 — MVP OK but dangerous

P2 — Acceptable post-launch

P3 — Cosmetic / polish

Justify every classification.

6) What Is the MINIMUM Fix to Reach “Real”?

For each major issue:

Describe the smallest possible implementation that makes it “real”

No ideal solutions — only minimum viable truth

Estimate:

effort (low / medium / high)

risk (low / medium / high)

7) Why This Dashboard Got THIS Percentage (Definitive Justification)

End each dashboard section with:

“This dashboard is rated XX% complete because…”

Then list exactly:

what is done

what is fake

what is missing

what blocks MVP

No marketing language. No optimism.

FINAL SECTION — CROSS-DASHBOARD TRUTH

After all dashboards are reviewed:

A) Which dashboards are safe to ship as-is?
B) Which dashboards must be limited or hidden at launch?
C) Which dashboard is the biggest illusion risk?
D) If only 3 things could be fixed in 14 days, what are they and why?
NON-NEGOTIABLE RULES

If something was not verified, say NOT VERIFIED

Do not repeat generic phrases like “needs improvement”

Every claim must reference:

UI behavior

backend logic

or data persistence

Assume this report decides whether the app ships

Output Format

Clear section headers

Tables where useful

No fluff

No repetition

Extreme clarity
