2️⃣ ⚠️ Wat KAN problemen geven (nu of later)
⚠️ 1. EXPO_PUBLIC_DOMAIN naam (niet fout, maar gevaarlijk)

Je setup werkt, maar:

EXPO_PUBLIC_DOMAIN = "glow-up-sports--ltvjeugd.replit.app"


En in code:

new URL(`https://${host}`)


🔴 Risico:

Iemand gebruikt per ongeluk EXPO_PUBLIC_DOMAIN zonder helper

Of plakt https:// er nog eens voor

Of gebruikt hem in WS / fetch direct

➡️ Dit is exact hoe bugs ontstaan (zoals je net had).

⚠️ 2. updates.enabled = false + runtimeVersion = appVersion

Niet fout, maar betekent:

Elke bugfix = nieuwe store build

Geen OTA hotfixes mogelijk

Voor MVP oké, maar:

❗ Zorg dat je deploy discipline strak is

⚠️ 3. Geen expliciete API URL

Je API is impliciet:

https://${EXPO_PUBLIC_DOMAIN}/api/...


Dat werkt, maar maakt je app:

minder leesbaar

minder schaalbaar (later aparte API host?)

3️⃣ 🔧 Verbeteringen die ik NU zou doen (sterk aanbevolen)
✅ VERBETERING 1 — hernoem domain env var (future-proof)

eas.json

"env": {
  "EXPO_PUBLIC_API_HOST": "glow-up-sports--ltvjeugd.replit.app",
  "EXPO_PUBLIC_ENV": "production"
}


en in code:

const host = process.env.EXPO_PUBLIC_API_HOST;

export const getApiUrl = (path: string) =>
  `https://${host}${path}`;


📌 Voordeel:

Geen twijfel meer

Iedereen weet: dit is de API

Minder kans op regressies

✅ VERBETERING 2 — één verplichte API helper (DO THIS)
// lib/api.ts
export function apiFetch(
  path: string,
  options?: RequestInit
) {
  return fetch(getApiUrl(path), {
    credentials: "include",
    ...options,
  });
}


🚫 Regel:

❌ fetch("/api/...") = BUG
✅ apiFetch("/api/...") = OK

✅ VERBETERING 3 — env guard (production safety)
if (!process.env.EXPO_PUBLIC_API_HOST) {
  throw new Error("Missing EXPO_PUBLIC_API_HOST");
}


➡️ Dit voorkomt “silent broken builds”.
