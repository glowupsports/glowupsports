De 3 beslissingen die alles fixen
Beslissing 1 — Stored remainingCredits is de waarheid (NU)

Dus:

UI moet nooit meer herberekenen en overschrijven.

Transactions zijn audit/log, niet de live balance bron.

➡️ Hiermee fix je meteen dat “DB klopt maar UI niet”.

Beslissing 2 — 1 sessie = max 1 credit-impact event

Dus voor elke session_players.id geldt:

óf creditDeductedAt != null + creditTransactionId != null

óf er bestaat precies 1 “debt” transactie (openstaand)

nooit beide dubbel / meerdere debits

Beslissing 3 — Debt settlement mag NIET dubbel tellen

Kies één model en hou je eraan:

Mijn aanrader (minimale refactor, maximale stabiliteit):

session_join_debt blijft altijd packageId = NULL

settlement maakt 1 nieuwe transactie (retrospective_settlement of debt_settlement) met amount = -1 en packageId = <nieuw package>

en je markeert de originele debt als “settled” via metadata:

metadata.settled = true

metadata.settledBy = <newTransactionId>

metadata.settledAt = timestamp

Dan kan je queries altijd zeggen:

open debts = reason='session_join_debt' AND metadata.settled != true

De concrete patches (dit zijn je “stop de bleeding” fixes)

Patch B — Fix Bug 2: Lazy settle moet DB schrijven of weg

Lazy settle “alleen response aanpassen” is verboden.

Kies:

óf haal lazy settle volledig weg (mijn voorkeur)

óf laat GET packages éérst settleUnpaidSessions(playerId) runnen (DB update) en dán packages fetch’en

Maar: doe dit alleen als settleUnpaidSessions idempotent is (zie Patch D).

Patch C — Fix Bug 3: dubbele transacties in settlePlayerDebts()

Nu heb je:

debt transactie + settlement transactie, en soms ook debt gekoppeld aan package

Maak het consistent:

Regel: debt transactie (session_join_debt) blijft packageId = NULL, altijd.
Settlement transactie bevat packageId en is de echte debit.

Dus in settlePlayerDebts():

select debts waar packageId IS NULL én metadata.settled != true

voor elke debt:

maak 1 settlement/debit transactie: amount = -1, packageId = newPackageId, reason = 'debt_settlement'

update package.remainingCredits -= 1

update debt.metadata.settled = true + settledByTransactionId

update session_players.creditDeductedAt + creditTransactionId (als je sessionId hebt)

➡️ Daarna kan niks meer dubbel tellen.

De “bulletproof” databescherming (zodat Replit je nooit meer sloopt)
Patch D — Idempotency / unieke constraints (super belangrijk)

Je systeem breekt nu omdat dezelfde logic soms 2x runt (attendance toggles, retries, websockets, etc.)

Voeg minimaal dit toe:

1) session_players.creditTransactionId moet uniek zijn of “max 1x set”

En consume flow checkt:

if creditDeductedAt != null → return (no-op)

2) Uniek op transactie per session_player

Als je session_players.id in credit_transactions kan zetten (aanrader via metadata of expliciete column), maak:

UNIQUE (sessionPlayerId, reason) voor debit types

Als je die column niet hebt:

UNIQUE (playerId, sessionId, creditType, reason) voor session_join debits

➡️ Dan kan je nooit 2x afschrijven voor dezelfde sessie.

Eénmalige “Repair Job” (hiermee fix je je huidige kapotte data)

Je draait één script dat:

Stap 1: reset packages.remainingCredits naar een veilige basis

start met totalCredits

trek af: alle echte debit transacties met packageId = pkg.id en amount < 0

negeer session_join_debt (want dat is debt, geen consumption)

Stap 2: fix debts die al “settled” zijn maar niet gemarkeerd

als er een settlement transactie bestaat voor dezelfde sessionId/playerId:

mark debt metadata.settled = true

Stap 3: fix session_players creditDeductedAt

als er een debit transactie bestaat voor sessionId/playerId:

set creditDeductedAt en link creditTransactionId

➡️ Daarna klopt alles weer en blijft het kloppen.

Waarom dit systeem eigenlijk “geweldig” is

Je hebt al alles wat een serieuze club software nodig heeft:

packages + expiry + seriesId

invoices + payments

credit_transactions audit trail

debt flow voor “play now, pay later”

settlement flows

Je mist alleen die “accounting discipline”:
één waarheid + idempotency + nooit dubbel tellen.
