Niceeee 🔥 goed moment dit — nu komt het fijne stuk: snel itereren zonder elke keer Play Store drama.

Omdat jij Expo + Play Store (test track) gebruikt, is Expo OTA Updates (EAS Update) exact wat je wil.

Ik leg het simpel + praktisch uit, zonder Expo-marketing-blabla.

TL;DR – zo werkt OTA bij jou

❌ NIET nodig: nieuwe Play Store upload voor elke UI/logic fix

✅ WEL nodig: app moet al 1x via Play Store geïnstalleerd zijn

Daarna:

UI fixes

API calls

business logic

feature flags
👉 via OTA in seconden live

Stap 1 — Check of je app OTA-ready is

In 90% van de Expo apps is dit al zo.

Check app.json / app.config.ts:

{
  "expo": {
    "updates": {
      "enabled": true,
      "checkAutomatically": "ON_LOAD",
      "fallbackToCacheTimeout": 0
    }
  }
}


fallbackToCacheTimeout: 0 = app start direct, checkt update op de achtergrond (beste UX)

Stap 2 — OTA updates publiceren (dit is het magische command)

Vanaf je project root:

npx expo install expo-updates


(Als je dit nog niet had)

Dan:

npx expo prebuild


(alleen nodig als je net expo-updates toevoegt)

Daarna OTA pushen:

npx expo update --branch production


👉 Binnen ±30 seconden:

testers openen app

update wordt automatisch geladen

geen Play Store nodig

Stap 3 — Branches slim gebruiken (heel belangrijk)

Gebruik OTA branches als “kanalen”.

Aanrader voor jou:

production → testers / live users

preview → jijzelf

dev → lokale tests

Voorbeeld:

npx expo update --branch preview


Dan:

alleen builds die op preview staan krijgen die update

Stap 4 — Zorg dat je Play Store build naar juiste branch luistert

In eas.json:

{
  "build": {
    "production": {
      "channel": "production"
    },
    "preview": {
      "channel": "preview"
    }
  }
}


En je Play Store test-build moet met:

npx expo build --profile production


of via EAS UI.

⚠️ Branch ≠ Channel, maar Expo koppelt ze automatisch als je dit goed zet.

Wat je WEL / NIET mag aanpassen via OTA (belangrijk voor Google)
✅ MAG via OTA

UI / styling

business logic

❌ MAG NIET via OTA

native code changes

permissions (camera, location, etc.)

app icon / splash

AndroidManifest wijzigingen

SDK upgrades die native modules veranderen

👉 Voor die dingen: nieuwe Play Store build nodig.

Pro tip (dit wil je echt)

Log OTA versies zodat je weet wie wat draait:

import * as Updates from "expo-updates";

console.log({
  updateId: Updates.updateId,
  channel: Updates.channel,
  runtimeVersion: Updates.runtimeVersion,
});


Super handig bij bug reports van testers.

Yep — that’s exactly how OTA feels to a tester: they open the app and suddenly the UI is different, without a Play Store update message. Because it didn’t come from Google Play… it came from Expo Updates.

If you don’t want it to “silently change” on people, you have a few clean options.

Why it happens

With Expo OTA, your app usually has something like:

checkAutomatically: "ON_LOAD" (or “ON_ERROR_RECOVERY”)

fallbackToCacheTimeout: 0 or low

So it checks for a new bundle and can apply it quickly (sometimes after a restart, sometimes same session depending on how you reload).

Option A (recommended): Show an in-app “Update available” banner

Instead of silent changes, you control when it applies.

Basic flow:

app starts → check update

if update exists → show banner: “Update available — Restart now”

user taps → reload app

Example logic:

import * as Updates from "expo-updates";

export async function checkAndPromptUpdate(setHasUpdate: (v:boolean)=>void) {
  try {
    const r = await Updates.checkForUpdateAsync();
    if (r.isAvailable) {
      await Updates.fetchUpdateAsync();
      setHasUpdate(true); // show UI button/banner
    }
  } catch {}
}

export async function applyUpdateNow() {
  await Updates.reloadAsync();
}


That fixes the “my UI changed randomly” feeling.

Option B: Only apply updates on next app restart (no mid-session surprise)

Keep background fetching, but don’t reload immediately.
User closes/reopens → they see the new version.

This is basically the “least annoying” silent approach.

Option C: Turn off automatic checking (strict control)

In app.json/app.config.ts:

"updates": {
  "enabled": true,
  "checkAutomatically": "NEVER"
}


Then you only check when you want (like in Settings → “Check for updates”).

Option D: Separate channels so testers don’t get “random experiments”

Use channels properly:

preview → you only

production → testers

So you can push wild UI changes to preview, and only push stable ones to production.
